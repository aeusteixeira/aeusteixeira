## **🧩 1\. Classe e Objeto**

Uma **classe** é como o molde — ela descreve como algo deve ser.  
 Um **objeto** é o que nasce a partir desse molde, a instância real.

class Carro {

    public $marca;

    public $cor;

    public function buzinar() {

        echo "Biiiii\!";

    }

}

$meuCarro \= new Carro();

$meuCarro-\>marca \= "Fiat";

$meuCarro-\>cor \= "Vermelho";

$meuCarro-\>buzinar(); // Biiiii\!

🧠 *Pense assim: a classe é a “receita”, o objeto é o “bolo pronto”.*

---

## **🏗️ 2\. Propriedades e Métodos**

As **propriedades** são os atributos (as “características”),  
 e os **métodos** são as ações (os “comportamentos”) da classe.

class Pessoa {

    public $nome;

    public $idade;

    public function apresentar() {

        echo "Olá, meu nome é {$this-\>nome} e tenho {$this-\>idade} anos.";

    }

}

$p \= new Pessoa();

$p-\>nome \= "Matheus";

$p-\>idade \= 26;

$p-\>apresentar();

---

## **🔐 3\. Encapsulamento**

Encapsulamento é o ato de **proteger os dados da classe**.  
 Em vez de deixar tudo aberto, você cria formas controladas de acessar e alterar os valores.

class ContaBancaria {

    private $saldo \= 0;

    public function depositar($valor) {

        if ($valor \> 0\) {

            $this-\>saldo \+= $valor;

        }

    }

    public function getSaldo() {

        return $this-\>saldo;

    }

}

$conta \= new ContaBancaria();

$conta-\>depositar(500);

echo $conta-\>getSaldo(); // 500

🧠 *Isso evita que alguém mude o saldo diretamente e quebre o sistema.*

---

## **🧬 4\. Herança**

Herança é quando uma classe **herda** características e comportamentos de outra.

class Animal {

    public function emitirSom() {

        echo "Som genérico";

    }

}

class Cachorro extends Animal {

    public function emitirSom() {

        echo "Au au\!";

    }

}

$cachorro \= new Cachorro();

$cachorro-\>emitirSom(); // Au au\!

🧩 *Um cachorro é um tipo de animal. Ele herda e pode modificar o comportamento.*

---

## **🎭 5\. Polimorfismo**

Polimorfismo significa “muitas formas”.  
 Ou seja: diferentes classes podem ter o **mesmo método**, mas **fazendo coisas diferentes**.

class Gato extends Animal {

    public function emitirSom() {

        echo "Miau\!";

    }

}

$animais \= \[new Cachorro(), new Gato()\];

foreach ($animais as $animal) {

    $animal-\>emitirSom();

}

🧠 *O mesmo método (`emitirSom`) muda o comportamento conforme o tipo do objeto.*

---

## **📜 6\. Abstração**

Uma classe **abstrata** serve como base para outras. Ela não pode ser usada diretamente, mas define uma estrutura para as classes filhas.

abstract class Forma {

    abstract public function calcularArea();

}

class Quadrado extends Forma {

    public $lado;

    public function \_\_construct($lado) {

        $this-\>lado \= $lado;

    }

    public function calcularArea() {

        return $this-\>lado \* $this-\>lado;

    }

}

$q \= new Quadrado(4);

echo $q-\>calcularArea(); // 16

🔹 *A classe abstrata diz o que deve existir, mas deixa as filhas decidirem como fazer.*

---

## **🔗 7\. Interfaces**

Uma **interface** define o que uma classe deve fazer, mas não como.  
 Ela é um contrato — quem implementa precisa cumprir.

interface Pagamento {

    public function pagar($valor);

}

class CartaoCredito implements Pagamento {

    public function pagar($valor) {

        echo "Pagando {$valor} com cartão de crédito.";

    }

}

class Pix implements Pagamento {

    public function pagar($valor) {

        echo "Pagando {$valor} via Pix.";

    }

}

🧠 *É como dizer: “Se você promete implementar Pagamento, precisa ter o método pagar()”.*

---

## **🧱 8\. Traits**

Uma **trait** é um jeito simples de compartilhar código entre várias classes sem precisar usar herança.

trait Loggable {

    public function log($mensagem) {

        echo "\[LOG\]: {$mensagem}";

    }

}

class Usuario {

    use Loggable;

}

$u \= new Usuario();

$u-\>log("Usuário criado."); // \[LOG\]: Usuário criado.

🔹 *Traits são úteis pra funções comuns, tipo logs, auditoria ou cache.*

---

## **🧰 9\. Namespaces e Autoload**

Namespaces servem pra **organizar o código** e evitar conflito entre classes com o mesmo nome.  
 O Composer usa isso pra fazer o **autoload automático** das classes.

namespace App\\Models;

class Usuario {

    public function \_\_construct() {

        echo "Classe Usuario carregada.";

    }

}

// Em outro arquivo:

use App\\Models\\Usuario;

$u \= new Usuario();

## **🧠 10\. SOLID (Princípios de Boas Práticas)**

| Letra | Nome | Ideia principal |
| ----- | ----- | ----- |
| **S** | Single Responsibility | Cada classe deve ter uma única responsabilidade. |
| **O** | Open/Closed | O código deve estar aberto pra extensão, mas fechado pra modificação. |
| **L** | Liskov Substitution | Classes filhas devem poder substituir a classe pai sem quebrar o sistema. |
| **I** | Interface Segregation | Melhor ter várias interfaces pequenas do que uma gigante. |
| **D** | Dependency Inversion | Dependa de abstrações, não de implementações diretas. |

---

## **⚙️ Exemplo completo com vários conceitos**

interface Notificacao {

    public function enviar($mensagem);

}

trait Log {

    public function registrar($msg) {

        echo "\[LOG\] {$msg}\\n";

    }

}

abstract class UsuarioBase {

    use Log;

    protected $nome;

    public function \_\_construct($nome) {

        $this-\>nome \= $nome;

    }

    abstract public function notificar($mensagem);

}

class UsuarioEmail extends UsuarioBase implements Notificacao {

    public function enviar($mensagem) {

        echo "Enviando e-mail para {$this-\>nome}: {$mensagem}\\n";

    }

    public function notificar($mensagem) {

        $this-\>registrar("Notificando {$this-\>nome}");

        $this-\>enviar($mensagem);

    }

}

$usuario \= new UsuarioEmail("Matheus");

$usuario-\>notificar("Bem-vindo ao sistema\!");

# **⚙️ Laravel – Conceitos que mais caem em entrevista**

---

## **🚦 Ciclo de Vida da Requisição**

1. Tudo começa no `public/index.php`;

2. O Laravel inicializa o **kernel HTTP**;

3. Passa pelos **middlewares**;

4. Resolve a rota;

5. Executa o controller;

6. E devolve a resposta ao navegador.

---

## **🧩 Service Container**

É o **coração do Laravel**.  
 Ele cuida da **injeção de dependência** — ou seja, cria e entrega automaticamente as classes que o sistema precisa.

class PedidoController {

    public function \_\_construct(PaymentService $payment) {

        $this-\>payment \= $payment;

    }

}

🧠 *Você não precisa dar “new PaymentService()”. O Laravel injeta pra você.*

---

## **🧱 Facades e Providers**

* **Facades:** acesso rápido a serviços registrados (`Cache::get()`, `Auth::user()`);

* **Providers:** onde os serviços são configurados (ex: `AppServiceProvider`).

---

## **🔐 Middlewares**

São filtros que interceptam a requisição antes e/ou depois do controller.  
 Exemplos: autenticação, logs, CORS.

---

## **🧩 Eloquent ORM**

Transforma tabelas do banco em **objetos PHP**.  
 Permite consultar dados de forma mais legível.

$users \= User::where('active', 1)-\>get();

🧠 *Faz a ponte entre o banco e o PHP sem precisar escrever SQL puro.*

---

## **🗄️ Banco de Dados e Relacionamentos**

* `first()` → retorna um registro.

* `get()` → retorna vários.

* `find($id)` → busca por ID.

* **Eager Loading** (`with()`): evita N+1 queries.

* **Relacionamentos:** `hasOne`, `hasMany`, `belongsTo`, `belongsToMany`.

* **Scopes:** filtros reutilizáveis dentro do model.

---

## **⚙️ Jobs, Queues e Events**

* **Jobs/Queues:** executam tarefas demoradas (ex: envio de e-mails) em background.

* **Events/Listeners:** permitem reagir a acontecimentos no sistema (ex: usuário cadastrado → enviar e-mail).

---

## **🔑 Auth, Gates e Policies**

* **Auth:** autenticação (quem é o usuário).

* **Gate:** autorização simples.

* **Policy:** regras por model (ex: só o dono pode editar o post).

* **API Tokens:** feitos com Laravel **Sanctum**.

---

## **🌐 API RESTful**

* `Route::apiResource()` cria rotas CRUD pra APIs.

* **FormRequest** → valida e autoriza os dados de entrada.

* **Handler.php** → onde você trata erros e exceções.

* **CORS:** controlado no middleware `HandleCors`.

---

## **🧠 Arquitetura e Boas Práticas**

* **Controller** → mínimo possível.

* **Service** → regras de negócio.

* **Repository** → camada de acesso ao banco.

* **Trait** → comportamento reutilizável.

* Aplique **SOLID** sempre que possível.

---

## **🧪 Testes**

* **Unit Test:** testa partes isoladas.

* **Feature Test:** testa o fluxo completo.

* Use **mocks/fakes** (`Mail::fake()`) pra simular dependências.

---

## **⚡ Performance e Deploy**

* Use `with()` pra evitar queries desnecessárias.

* Adicione **índices** no banco.

* Use cache com `Cache::remember()`.

* Automatize tarefas com o **scheduler** no `Kernel.php`.

* No deploy: configure `.env`, rode `migrate --force`, e ative queues.

## **🔍 Debug e Monitoramento**

* Veja erros em `storage/logs/laravel.log`.

* Ferramentas úteis: **Laravel Telescope**, **Sentry**, **Bugsnag**.

* Pra debug local: use `dump()` ou `dd()`.

## **este Unitário**

**Foca na menor parte possível do sistema** — geralmente uma função, método ou classe isolada.

* Testa **uma única unidade** de lógica.

* Não depende de banco, API, view, nada externo.

* É rápido e serve para garantir que a regra de negócio funciona sozinha.

* Ideal para validar cálculos, validações, decisões, helpers etc.

**Exemplo:**  
 Testar se um método `calcularDesconto()` retorna o valor correto.

---

## **✅ Teste Funcional**

**Foca no comportamento completo de uma funcionalidade**, simulando o uso real do sistema.

* Envolve várias partes trabalhando juntas.

* Pode bater em rotas, usar banco (via refresh), middleware, autenticação.

* Testa o fluxo como o usuário final utiliza.

* Serve para garantir que o sistema está funcionando como um conjunto.

**Exemplo:**  
 Enviar uma requisição POST para `/login` e verificar se o usuário é autenticado corretamente.

## **Resumo – Mensageria: Kafka, RabbitMQ e Jobs no Laravel**

**Kafka e RabbitMQ são soluções de mensageria de infraestrutura; os Jobs do Laravel são uma abstração de aplicação.**

**RabbitMQ é uma fila de mensagens tradicional, orientada a work queues.**  
**É ideal para tarefas assíncronas com roteamento, retries, dead-letter e baixa latência, como envio de e-mails, integrações, processamento de pagamentos e workflows entre sistemas.**

**Kafka é um sistema de event streaming baseado em log.**  
**É indicado para alto volume de eventos, onde as mensagens precisam ser retidas, reprocessadas e consumidas por múltiplos serviços, como eventos de domínio, auditoria, pipelines de dados e integrações em larga escala.**

**Jobs/Queues do Laravel não são um broker, mas uma camada de orquestração da aplicação.**  
**Eles representam a unidade de trabalho e dependem de um backend (Redis, RabbitMQ, SQS, etc.).**  
**São usados para desacoplar tarefas do request, melhorar performance e organizar processamento assíncrono dentro do sistema.**

**Resumo prático:**

* **Tarefa assíncrona de aplicação → Laravel Jobs (geralmente com Redis)**  
* **Workflows e integração entre serviços → RabbitMQ**  
* **Eventos em escala, com retenção e reprocessamento → Kafka**
